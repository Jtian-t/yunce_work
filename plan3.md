# 招聘协同平台 V3 方案

## Summary

- V3 重点解决当前“简历解析效果差、候选人基础信息不可直接编辑、面试推进链路不完整、部门与面试官协作视角不足”的问题。
- 本版不推翻 V2 的 MySQL + MinIO、解析任务、辅助决策任务基础，而是在其上升级成“更像真实招聘系统”的可用版本。
- 核心目标分为三块：
  - 简历解析升级：从当前轻量规则抽取升级为“文本抽取 + OCR + 版面理解 + 结构化映射 + 人工纠偏”的组合链路
  - 候选人详情升级：基础信息支持直接编辑，解析结果和人工信息可协同维护
  - 流程协同升级：面试安排携带会议链接、每一面单独沉淀反馈、部门和面试官都能看到自己该看的内容并收到通知

## Current Problems

### 1. 简历解析效果不足

- 当前 `LoggingAgentDispatcher` 的解析本质上仍是规则型 mock 实现，适合演示，不适合真实复杂简历。
- 主要问题：
  - 对多栏 PDF、扫描版 PDF、图片型 PDF 兼容性差
  - 技能提取主要依赖关键词命中，缺乏上下文和版面理解
  - 项目经历无法稳定按项目切分，容易把整段文字混在一起
  - 最近公司、岗位、工作年限、教育信息等字段缺乏稳定定位能力
  - 没有把“文本层 PDF”和“扫描型 PDF”分开处理

### 2. 候选人基础信息不可直接修改

- 当前详情页更偏展示页
- 基础信息如手机号、邮箱、城市、学历、经验、技能摘要、项目摘要缺少直观编辑入口
- 解析结果无法方便地回填后再人工修正

### 3. 面试推进链路仍不完整

- 安排面试时没有会议链接字段
- 面试通知链路不完整，面试官无法收到明确属于自己的面试任务和会议地址
- 每一面对应的评价虽有数据模型，但前端展示和部门视角不够完整
- 部门侧缺少“按轮次查看候选人进展、面试反馈、下一步动作”的统一视图

## V3 Goals

### 目标一：把简历解析从“演示可用”升级为“业务可用”

- 能区分文本型 PDF 与扫描型 PDF
- 能较稳定提取：
  - 姓名
  - 手机号
  - 邮箱
  - 城市
  - 学历
  - 工作年限
  - 最近公司 / 最近岗位
  - 技能标签
  - 项目经历列表
  - 自我评价 / 核心亮点
- 项目经历要按“项目名称 / 时间 / 角色 / 技术栈 / 职责 / 成果摘要”拆分，而不是大段拼接
- 支持人工修正后保留结果，不被低质量自动解析覆盖

### 目标二：候选人详情页支持“展示 + 编辑 + 回填”

- 增加“编辑基础信息”按钮
- 支持弹窗或侧边抽屉编辑候选人基础信息
- 支持将解析结果按字段一键回填到表单
- 支持“自动解析建议值”和“人工最终值”同时保留

### 目标三：让面试流程真正闭环

- 安排面试时支持会议链接
- 默认支持“腾讯会议 / 腾讯视频会议链接”字段
- 后续如有企业微信或腾讯会议开放平台凭证，可升级为自动创建会议
- 面试官收到自己的面试通知，通知中包含：
  - 候选人姓名
  - 岗位
  - 面试轮次
  - 时间
  - 腾讯会议链接
  - 候选人简历入口
- 每一面都要有独立评价记录和展示区域

### 目标四：补齐部门视角

- 部门负责人除了“部门筛选”外，还要能看到：
  - 候选人在本部门的处理进展
  - 已安排的面试轮次
  - 每一轮的面试反馈
  - 当前建议动作
- 部门页支持从“待反馈”扩展为“部门候选人工作台”

## PDF / 简历解析技术路线建议

## 一、当前常用的 PDF 解析技术分类

### 1. 文本层 PDF 直接抽取

适合：

- 可复制文本的普通 PDF
- 招聘网站导出的标准简历 PDF

常用技术：

- Apache PDFBox（Java，适合当前后端）
- PyMuPDF（Python，文本抽取和版面信息能力强）
- Unstructured 的 fast / hi_res 分区方案

特点：

- 成本低
- 速度快
- 对文本型 PDF 很有效
- 对扫描件无能为力

### 2. OCR 识别扫描版 PDF

适合：

- 图片型 PDF
- 手机拍照上传
- 扫描版简历

常用技术：

- Tesseract OCR
- PaddleOCR
- Azure Document Intelligence
- Google Document AI

特点：

- 能处理扫描件
- 但仅有 OCR 还不够，通常还需要版面理解

### 3. 版面理解 / 文档结构化解析

适合：

- 多栏简历
- 模板复杂简历
- 需要按标题、段落、项目块切分的场景

常用技术：

- Unstructured
- Azure Document Intelligence Layout
- Google Document AI Layout / Form Parser
- MinerU 一类面向复杂 PDF 结构化提取的方案

特点：

- 能识别段落、表格、标题、块结构
- 更适合“技能”和“项目经历”的稳定切分

### 4. LLM / VLM 结构化抽取

适合：

- 需要从自然语言段落里提炼结构字段
- 需要把技能、项目、亮点、风险做更接近 HR 语义的整理

特点：

- 对复杂语义抽取更强
- 成本比规则和 OCR 高
- 更适合作为第二阶段“结构化归纳器”，不适合单独承担原始 PDF 读取

## 二、针对本项目的推荐路线

不建议继续只靠单一 PDFBox + 正则。

推荐改成四层链路：

1. 文档接入层
   - 先判断 PDF 是否存在可抽取文本层
   - 有文本层：优先直接抽文本
   - 无文本层：转图片后走 OCR

2. 版面切分层
   - 识别标题、段落、列表、项目块、技能块
   - 对“项目经历”“工作经历”“技能清单”等模块单独切片

3. 结构抽取层
   - 用规则 + 模型结合方式提取基础字段
   - 对技能、项目经历、最近岗位等重点字段单独抽取

4. 结果归一层
   - 输出统一结构
   - 每个字段保留：
     - value
     - confidence
     - source
     - rawSnippet

## V3 Functional Scope

### 1. 简历解析模块升级

#### 1.1 新解析架构

- 新增 `ResumeExtractionPipeline`
- 按阶段拆分：
  - `PdfTextExtractor`
  - `PdfOcrExtractor`
  - `ResumeLayoutSegmenter`
  - `ResumeStructuredFieldExtractor`
  - `ResumeProjectExtractor`
  - `ResumeSkillExtractor`
  - `ResumeNormalizationService`

#### 1.2 增强字段模型

- 从“扁平字段”升级为更细粒度结构：
  - 基础字段
  - 技能标签数组
  - 项目经历数组
  - 工作经历数组
  - 教育经历数组
  - 高亮摘要
  - 风险 / 缺失项

#### 1.3 项目经历识别升级

- 项目经历拆成：
  - `projectName`
  - `period`
  - `role`
  - `techStack`
  - `responsibilities`
  - `achievements`
  - `summary`

#### 1.4 技能识别升级

- 技能从单个长字符串升级为标签集合
- 支持：
  - 原始命中词
  - 标准化技能名
  - 来源片段
  - 置信度

#### 1.5 人工修正机制

- 解析结果只作为建议值
- 人工确认值单独存储
- 新一轮解析不能自动覆盖已人工锁定字段

### 2. 候选人基础信息编辑

#### 2.1 详情页新增“编辑基础信息”按钮

- 位置放在候选人头像卡片或基础信息卡片右上角
- 点击后弹出 `Dialog` 或 `Drawer`

#### 2.2 可编辑字段

- 姓名
- 电话
- 邮箱
- 城市
- 岗位
- 学历
- 工作年限
- 技能摘要
- 项目摘要
- 来源
- 下一步动作

#### 2.3 回填交互

- 解析结果中对高置信字段提供“一键回填”
- 回填后允许再手工修改
- 保存后立即刷新详情页

### 3. 面试安排与流程推进升级

#### 3.1 面试安排字段补齐

- `InterviewPlan` 新增：
  - `meetingType`
  - `meetingUrl`
  - `meetingId`
  - `meetingPassword`
  - `interviewStageCode`
  - `interviewStageLabel`
  - `organizerId`
  - `departmentId`

#### 3.2 安排面试表单升级

- 新增字段：
  - 面试轮次
  - 面试类型（现场 / 腾讯会议 / 电话）
  - 腾讯会议链接
  - 会议号
  - 密码
  - 面试备注

#### 3.3 腾讯会议对接策略

V3 建议分两阶段：

- 第一阶段：
  - 先支持手工填写腾讯会议链接
  - 通知和详情页完整展示
- 第二阶段：
  - 如果后续有腾讯会议开放平台凭证，再做自动创建会议

### 4. 面试反馈与多轮展示升级

#### 4.1 每一面单独反馈

- 每条 `InterviewPlan` 下面展示独立评价列表
- 每轮反馈显示：
  - 面试官
  - 结果
  - 分数
  - 优势
  - 风险
  - 建议
  - 提交时间

#### 4.2 多轮面试时间线

- 在候选人详情页新增“面试进展”卡片
- 清晰展示：
  - 一面已安排 / 已完成 / 已评价
  - 二面已安排 / 已完成 / 已评价
  - HR 面 / 终面状态

#### 4.3 推进规则升级

- 只有前一轮面试完成后，才允许推进下一轮或直接推进 Offer
- 若存在未提交的面试评价，HR 推进时给出提示

### 5. 部门视角升级

#### 5.1 部门工作台

- 新增或升级部门页面为：
  - 待部门筛选
  - 部门处理中
  - 本部门待面试
  - 本部门面试中
  - 本部门已完成

#### 5.2 部门视角可见内容

- 候选人基础信息
- 简历预览
- 简历解析摘要
- 本部门反馈历史
- 本部门相关面试轮次
- 每轮面试结果
- 当前建议动作

### 6. 面试官视角与通知升级

#### 6.1 面试官待办页

- 新增“我的面试”页面
- 只展示分配给当前面试官的面试安排

字段包括：

- 候选人姓名
- 岗位
- 面试轮次
- 面试时间
- 腾讯会议链接
- 候选人简历
- 面试评价是否已提交

#### 6.2 通知机制

- 当前版本先走站内通知
- 每次新建面试安排时，为对应面试官创建通知
- 通知内容包含：
  - 候选人姓名
  - 轮次
  - 时间
  - 会议链接
  - 快速进入详情页链接

#### 6.3 后续可扩展通知渠道

- 邮件
- 企业微信
- 腾讯会议日历集成

- 候选人主表保留人工最终字段
- 新增或增强解析结果表，存结构化数组和字段来源

### 面试安排相关

- `InterviewPlan`
  - 新增会议信息字段
  - 新增轮次编码字段
  - 新增部门归属字段

### 通知相关

- `Notification`
  - 增加通知类型：`INTERVIEW_ASSIGNED`
  - 增加扩展 JSON，用于保存会议链接、候选人 ID、面试 ID

### 解析结果相关

- 新增 JSON 字段：
  - `skillsJson`
  - `projectsJson`
  - `experiencesJson`
  - `educationsJson`
  - `rawBlocksJson`

## API Design

### 简历解析

- `POST /api/candidates/{id}/parse-jobs`
- `GET /api/candidates/{id}/parse-jobs/latest`
- `POST /api/candidates/{id}/parsed-profile/apply`
- `PUT /api/candidates/{id}/parsed-profile/manual-fields`

### 候选人基础信息

- `PUT /api/candidates/{id}`
  - 支持详情页直接编辑基础信息

### 面试安排

- `POST /api/interviews`
  - 扩展请求字段，支持会议链接
- `GET /api/interviews/mine`
  - 当前面试官的面试列表

### 面试反馈

- `POST /api/interviews/{id}/evaluations`
- `GET /api/candidates/{id}/interviews`
  - 返回每轮评价详情

### 通知

- `GET /api/notifications`
- `POST /api/notifications/{id}/read`

## Implementation Phases

### Phase 1：先解决易用性问题

- 候选人基础信息编辑
- 面试安排表单补会议链接
- 面试官通知补齐
- 每轮面试反馈展示

### Phase 2：升级简历解析质量

- 区分文本 PDF 和扫描 PDF
- 增强技能与项目分解
- 增强结构化结果模型
- 加入人工修正和回填

### Phase 3：完善部门 / 面试官工作台

- 部门工作台
- 我的面试
- 我的通知
- 轮次状态和推进校验

## Acceptance Criteria

### 简历解析

- 文本型 PDF 能稳定提取技能标签和项目列表
- 扫描型 PDF 至少能通过 OCR 提取基础文本
- 项目经历不再只显示零散句子，能够按项目块展示

### 基础信息编辑

- 详情页能直接编辑并保存候选人基础信息
- 保存后列表和详情同步更新

### 面试流程

- 安排面试时可填写腾讯会议链接
- 面试官能在自己的待办中看到链接
- 候选人详情页按每一面显示反馈

### 部门视角

- 部门人员能看到本部门候选人进展
- 能查看本部门面试结果和下一步动作

## Risks

- 真正提升 PDF 简历解析效果，不能只改前端展示，必须改解析链路本身
- 若要做自动创建腾讯会议，需要额外的第三方平台凭证和权限
- 如果继续只保留 Java 单链路，OCR 与版面理解能力提升有限，建议预留 Python 解析服务扩展点

## Recommendation

V3 建议采用“主系统仍保留 Java Spring Boot，复杂简历解析引入独立解析服务”的架构：

- 主系统：候选人、流程、通知、权限、展示
- 解析服务：PDF 文本提取、OCR、版面切分、结构化抽取

这样能最大化降低对现有业务系统的侵入，同时显著提升简历解析质量。
